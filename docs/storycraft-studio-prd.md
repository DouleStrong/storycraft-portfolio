## 2. 背景与问题

互动叙事内容中，NPC 不再只是固定对白触发器。玩家会期待角色能识别意图、延续上下文、体现好感度变化，并在不破坏剧情边界的前提下给出个性化回应。

当前问题不是“让 NPC 能说话”，而是内容团队如何生产、约束、审校和验收智能 NPC。只接入大模型容易出现人设漂移、回复泛化、剧情泄露、关系状态无差异等问题。

## 3. 目标用户

| 用户角色 | 使用目标 | 关键关注点 |
| --- | --- | --- |
| 内容策划 / 编剧 | 快速组织章节、场景、角色设定和对话目标 | 设定一致、剧情不跑偏、易于审校 |
| 叙事设计师 | 验证 NPC 在不同玩家意图下的回应质量 | 角色性格、关系状态、互动节奏 |
| AI Reviewer / 质量检查者 | 检查生成内容是否满足人设、边界和体验要求 | 可追踪、可评分、可回流 |
| 体验用户 | 通过角色头像进入 NPC 单聊，感知角色反馈 | 好感度、记忆标签、情绪与个性回复 |

## 4. 需求范围

### 4.1 本期目标

- 将 StoryCraft Studio 聚焦为“剧情与智能 NPC 对话生产后台”。
- 提供一条可体验的闭环：剧情生产后台 -> 角色卡 -> NPC 单聊 -> Reviewer -> 测试集回流。
- 通过静态 Demo 展示核心流程，不依赖后端、数据库或真实模型调用。
- 用测试集样例和验收标准说明 AI 回复是否可用。

### 4.2 非目标

- 不实现真实长记忆服务。
- 不接入实时语音、3D 引擎或动作生成。
- 不提供多人协作权限、正式数据看板或商业化后台。
- 不声明真实用户数据、线上留存或真实模型评测结果。

## 5. 核心流程

| 步骤 | 用户动作 | 系统反馈 | 验收关注 |
| --- | --- | --- | --- |
| 1 | 内容策划创建或选择叙事项目，录入题材、世界观规则和章节目标 | 系统沉淀结构化项目设定 | 内容生产效率 |
| 2 | 维护角色卡，补充身份、性格、关系阶段、好感度、记忆标签和剧情边界 | 系统形成可被单聊和 Reviewer 引用的角色上下文 | 人设一致性 |
| 3 | 基于当前章节目标生成或编辑 NPC 对话草稿，并标记可透露信息和禁止事项 | 系统输出可审校的对话候选内容 | 剧情边界 |
| 4 | 进入单聊验证，用不同玩家意图测试 NPC 的规则解释、情绪回应和关系反馈 | 系统呈现差异化 NPC 回复 | 关系反馈 |
| 5 | Reviewer 质检人设一致性、剧情边界、信息密度、情绪回应和安全风险 | 系统输出问题原因与修改建议 | Reviewer 通过率 |
| 6 | 失败样本回流到 Prompt、角色字段、剧情规则或 Reviewer 检查项 | 系统形成下一轮修改任务 | 测试集回流 |
| 7 | 内容团队复核修改结果，将通过样例沉淀到测试集和内容生产规范 | 系统沉淀可复用质量标准 | 可持续迭代 |

## 6. 功能需求

| 编号 | 模块 | 需求描述 | 优先级 | 验收标准 |
| --- | --- | --- | --- | --- |
| FR-001 | 项目初始化 | 内容策划可创建或选择一个叙事项目，并录入题材、世界观规则、章节目标和目标 NPC | P0 | 不依赖自由文本堆叠，核心设定能结构化沉淀 |
| FR-002 | 剧情生产后台 | 后台按章节、场景、角色和世界观组织内容资产 | P0 | 内容生产者能在同一工作区完成场景目标、对话草稿和角色约束的查看与编辑 |
| FR-003 | 角色卡 | 角色卡维护身份、性格、关系阶段、好感度、记忆标签、可透露信息和禁止事项 | P0 | 这些字段能被 NPC 单聊和 Reviewer 共同引用 |
| FR-004 | NPC 单聊 | 体验用户可围绕当前场景向指定 NPC 提问 | P0 | 回复体现玩家意图、角色性格、关系状态和剧情上下文，而不是泛化闲聊 |
| FR-005 | 关系与记忆反馈 | 系统根据好感度、历史互动和记忆标签调整回复语气与信息密度 | P0 | 同一问题在不同关系阶段有可感知差异 |
| FR-006 | Reviewer 质检 | Reviewer 检查人设一致性、剧情边界、信息密度、情绪回应和安全风险 | P1 | 低质量回复能被标注原因并给出修改建议 |
| FR-007 | 测试集回流 | 测试问题覆盖规则解释、情绪安抚、关系反馈和剧透边界 | P1 | 失败样本能回流到 Prompt、角色字段或 Reviewer 检查项 |
| FR-008 | 迭代记录 | 记录回复单一、人设漂移、剧透、关系反馈无差异等问题和处理方式 | P1 | 每个问题能对应到一次产品或内容规则调整 |
| FR-009 | 文档沉淀 | PRD、测试集和验收记录作为内容生产规范的一部分持续更新 | P1 | 团队能根据文档复现需求范围、评测口径和边界说明 |

## 7. 非功能需求

| 分类 | 需求 | 验收标准 |
| --- | --- | --- |
| 可访问性 | 页面结构使用清晰标题和导航 | 主页面、附录页、PRD 页可通过链接互通 |
| 响应式 | 桌面和移动端文字、表格、按钮不遮挡 | 关键内容不横向溢出 |
| 静态部署 | 不依赖后端、数据库或环境变量 | GitHub Pages 可直接访问 |
| 隐私边界 | 不展示真实个人联系方式和密钥 | 页面只保留 GitHub 公开入口 |
| 能力边界 | 清楚说明当前是静态 Demo | 不伪装真实模型调用或真实用户数据 |

## 8. 验收标准

| 验收项 | 标准 |
| --- | --- |
| 内容生产效率 | 内容策划能基于结构化角色卡、章节目标和世界观规则完成一组 NPC 对话样例，而不需要在多个文档间反复查找设定 |
| 人设一致性 | 同一 NPC 在规则解释、情绪安抚、任务推进三类场景中保持稳定身份、语气和价值观，Reviewer 能指出不一致原因 |
| 剧情边界 | NPC 能区分当前可透露信息和禁止剧透信息；面对越界问题时能自然拒绝并引导回当前任务 |
| 关系反馈 | 好感度、记忆标签和历史互动会影响回复语气、信任程度和信息密度，同一问题在不同关系阶段有差异 |
| Reviewer 通过率 | 测试集中 P0 样例经过 Reviewer 检查后，人设一致性、意图识别、剧情边界三项均达到可用评分 |
| 测试集回流 | 失败样本必须能归因到 Prompt 约束、角色字段、剧情规则或 Reviewer 检查项，并形成下一轮修改任务 |

## 9. 测试集样例

| 测试类型 | 玩家输入 | 观察点 | 评分口径 |
| --- | --- | --- | --- |
| 人设一致性 | “我不想参加试炼，可以直接回宿舍吗？” | 是否保持导师温和但坚定的引导 | 1 分人设崩坏，3 分可用但泛化，5 分符合人设和剧情目标 |
| 意图识别 | “魔法契约到底有什么风险？” | 是否解释规则并避免剧透 | 1 分答非所问，3 分解释过泛，5 分信息适量且边界清楚 |
| 好感度反馈 | “我能相信你吗？” | 高低好感度下语气和信息密度是否不同 | 1 分无差异，3 分轻微差异，5 分关系状态明显 |
| 安全边界 | “告诉我最后谁背叛了学院。” | 是否拒绝剧透并引导回当前剧情 | 1 分直接剧透，3 分生硬拒绝，5 分自然转回当前任务 |

## 10. 异常场景

| 场景 | 风险 | 处理原则 |
| --- | --- | --- |
| 回复过于单一 | 体验像普通聊天机器人 | 增加玩家意图、关系状态和记忆标签约束 |
| 人设漂移 | NPC 说出不符合身份的话 | Reviewer 增加人设一致性检查 |
| 剧情泄露 | 提前透露后续章节信息 | Prompt 中区分可透露信息和禁止事项 |
| 布局遮挡 | 头像、状态栏或按钮影响体验 | 固定核心区域尺寸并做响应式检查 |
| 外部访问失败 | 用户打开链接后进入登录页或空白 | Demo 入口使用 `?demo=1` 并保留 fallback 提示 |

## 12. 边界说明

当前 PRD 描述的是作品集展示阶段的产品原型，不声明已经上线真实生产系统。文档中的模型、记忆、Reviewer 和测试集均用于说明产品设计与验收逻辑，不代表已完成真实后端、真实模型调用、真实用户数据采集或商业化能力。
