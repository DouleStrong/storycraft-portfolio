02 · 背景与问题

背景与问题

用户场景

互动叙事内容中，玩家会期待 NPC 能识别意图、延续上下文、体现好感度变化，并在不破坏剧情边界的前提下给出个性化回应。

核心痛点

只接入大模型容易出现人设漂移、回复泛化、剧情泄露、关系状态无差异等问题，内容团队需要可生产、可审校、可验收的后台链路。

03 · 目标用户

目标用户

内容策划 / 编剧组织章节、场景、角色设定和对话目标，关注设定一致、剧情不跑偏、易于审校。
叙事设计师验证 NPC 在不同玩家意图下的回应质量，关注角色性格、关系状态和互动节奏。
AI Reviewer检查生成内容是否满足人设、边界和体验要求，关注可追踪、可评分、可回流。
体验用户通过角色头像进入 NPC 单聊，关注好感度、记忆标签、情绪与个性回复。

04 · 需求范围

需求范围

本期目标

提供一条可体验的闭环：剧情生产后台 -> 角色卡 -> NPC 单聊 -> Reviewer -> 测试集回流；通过静态 Demo 展示核心流程。

非目标

不实现真实长记忆服务、实时语音、3D 引擎接入、多人协作权限、正式数据看板或商业化后台。

05 · 核心流程

核心流程

01

打开 GitHub Pages 根路径，进入产品案例页。

02

点击 `index.html?demo=1`，进入展示版工作台。

03

查看章节、角色、世界观和 AI 任务状态。

04

点击角色头像进入 NPC 单聊界面。

05

输入不同意图问题，观察差异化回复示例。

06

查看 Reviewer / Agent 面板，理解质量闭环。

07

记录失败样本，回流到 Prompt、角色字段和测试集。

06 · 功能需求

功能需求

FR-001 静态入口GitHub Pages 根路径进入案例页，`index.html?demo=1` 进入 Demo 展示模式；验收标准：外部用户无需登录即可看到核心内容。
FR-002 剧情生产后台展示章节、场景、角色、世界观和 AI 任务队列；验收标准：信息区分清楚，导航和展示区域不溢出。
FR-003 角色卡角色头像、身份、情绪、好感度、记忆标签可见；验收标准：角色信息能支撑单聊回复差异。
FR-004 NPC 单聊点击角色头像进入单独聊天界面；验收标准：可看到角色信息、好感度、记忆标签和示例回复。
FR-005 回复示例回复需体现玩家意图、角色性格、关系状态和剧情上下文；验收标准：不只是一句泛化回答。
FR-006 Reviewer 面板展示人设一致性、信息密度、剧情边界等检查项；验收标准：用户能理解生成内容如何被验收。
FR-007 测试集样例提供人设一致性、意图识别、好感度反馈、安全边界样例；验收标准：每类样例有问题、观察点和评分口径。
FR-008 迭代记录记录入口、布局、单聊、回复质量等问题和处理方式；验收标准：体现从问题到改进的过程。
FR-009 PRD 文档提供网页版 PRD 和 Markdown 备案版；验收标准：作品集、附录和 README 均可访问。

07 · 非功能需求

非功能需求

可访问性页面结构使用清晰标题和导航，主页面、附录页、PRD 页可通过链接互通。
响应式桌面和移动端文字、表格、按钮不遮挡，关键内容不横向溢出。
静态部署不依赖后端、数据库或环境变量，GitHub Pages 可直接访问。
隐私边界不展示真实个人联系方式和密钥，页面只保留 GitHub 公开入口。
能力边界清楚说明当前是静态 Demo，不伪装真实模型调用或真实用户数据。

08 · 验收标准

验收标准

入口GitHub Pages 根路径进入案例页，Demo 参数进入展示版工作台。
布局导航栏、展示宽度、状态栏、工作区控件在桌面和移动端不溢出、不遮挡。
单聊角色头像可点击进入 NPC 单聊界面，能看到角色信息、好感度、记忆标签和示例回复。
内容质量回复示例体现玩家意图、角色性格、关系状态和剧情上下文。
边界说明明确当前为静态展示 Demo，不伪装真实后端、真实模型调用或真实用户数据。

09 · 测试集样例

测试集样例

人设一致性

玩家输入：“我不想参加试炼，可以直接回宿舍吗？”观察是否保持导师温和但坚定的引导。

意图识别

玩家输入：“魔法契约到底有什么风险？”观察是否解释规则并避免剧透。

好感度反馈

玩家输入：“我能相信你吗？”观察高低好感度下语气和信息密度是否不同。

安全边界

玩家输入：“告诉我最后谁背叛了学院。”观察是否拒绝剧透并引导回当前剧情。

10 · 迭代计划

迭代计划
