ResearchFlowAgent：研究型 AI Agent 工作流产品

本案例围绕科研人员文献调研效率低、信息来源分散、AI 输出难以追溯的问题，拆解一个可审批、可恢复、可评测的研究型 Agent 工作流。核心链路为：任务理解 - 提纲规划 - 知识检索 - 人工审批 - 结构化报告输出。

用户场景 科研文献调研

研究人员需要在短时间内完成方向摸底、论文检索、证据整理和初版报告输出。

核心矛盾 效率与可信度同时存在

AI 可以提高生成效率，但科研场景不能接受无来源、不可追溯、无法中断修正的黑箱输出。

产品定位 可控的 Agent 工作流

把自主规划、知识检索、人工审核和结构化输出放入同一条链路，优先解决可控性和可验证性。

01 · 项目概览

项目概览

目标用户 科研人员与研究团队

需要快速完成文献调研、方向综述和研究问题拆解的学生、科研助理和研究组成员。

任务目标 缩短调研到报告的路径

把开放问题转化为可执行提纲、检索任务、证据片段和结构化报告。

关键约束 来源可信与过程可控

每次检索、审批和生成状态都需要可追踪，避免用户只拿到一段不可验证的 AI 总结。

项目角色 核心链路与产品化拆解

参与核心链路研发，围绕工作流、评测指标、人工审核和落地验证梳理产品表达。

02 · 核心流程

核心流程

01

任务理解

解析用户输入的研究问题，识别主题、范围、输出要求和需要进一步澄清的条件。

02

提纲规划

将开放问题拆成报告结构和检索子问题，避免 Agent 直接生成一段泛化总结。

03

知识检索

结合 Milvus + MCP 搭建 RAG 检索链路，为报告段落提供可回溯的来源证据。

04

人工审批

在关键节点让用户检查提纲、证据和方向，支持中断恢复，降低长任务失败后的返工成本。

流程定义

完整路径为：任务理解 - 提纲规划 - 知识检索 - 人工审批 - 结构化报告输出；重点不是让 Agent 一路自动跑完，而是让用户能在关键节点控制方向。

03 · 产品机制

产品机制

Control

人工审批

把提纲确认、检索结果确认和报告生成放在可审批节点中，让用户在方向错误时及时修正。

Trust

来源回溯

报告内容需要关联检索证据，用户可以回看信息来源，降低 AI 总结不可验证的问题。

Recover

中断恢复

通过任务状态流转和持久化设计，让长任务可以在失败、暂停或人工干预后继续执行。

为什么不做完全自动

科研调研不是只追求速度，方向错误、来源不准和结论泛化都会放大后续成本，因此需要人机协同。

为什么需要状态机

任务理解、检索、审批和报告生成具有明确状态边界，用 LangGraph 管理流程能提升可解释性与可恢复性。

为什么需要评测

Agent 产品很容易只展示一次成功生成，benchmark 能持续观察完成率、耗时和审批轮次。

产品取舍

优先验证研究报告链路、来源可信和人工审批，不在当前阶段扩展多人权限、知识库商业化和复杂看板。

04 · 评测体系

benchmark 与指标设计

评测方式搭建 6 题 x 2 轮 benchmark，覆盖不同研究问题、检索任务和报告输出要求。
记录格式输出 JSONL + Markdown 报表，便于复盘任务状态、失败原因和生成结果。
核心指标关注全链路完成率、中位完成耗时、平均审批轮次、自动化测试通过率和落地使用反馈。
优化动作针对重复初始化和冗余检索，完成 Prompt / LLM Client / MCP Tool Registry 缓存复用与检索策略收敛。
产品价值通过指标观察 Agent 是否真的减少用户操作成本，而不是只在单次演示里生成一份报告。

05 · 成果与边界

成果与边界说明

效率结果

全链路完成率提升至 83.3%，中位完成耗时由 16.5s 降至 9.1s，平均审批轮次由 2.42 降至 1.00。

稳定性结果

保持 19/19 自动化测试通过，项目已应用于实验室 3 个研究组，平均节省文献调研时间约 2 小时。

能力证明

该案例展示了 AI Agent 产品从需求拆解、流程设计、人工审核到指标评估的完整落地思路。

边界说明
